## Шаг 1. Сегмент, сценарий и целевая «первая победа»

### 1.1 Beachhead-сегмент (точка входа)

| Параметр | Значение |
|----------|----------|
| **Кто** | Хосты регулярных компаний друзей **18–35 лет** в Москве и СПб, играющие в мафию ≥ 1 раз в месяц офлайн (дома, в антикафе) или онлайн (удалённо) |
| **Где** | (1) Дома, на даче, (2) в антикафе, (3) удалённо через созвон. Веб-приложение работает во всех трёх сценариях — нужен только смартфон или ноутбук |
| **Почему именно они** | (1) активная аудитория Telegram / ВКонтакте, (2) платёжеспособная (студенты старших курсов и работающая молодёжь), (3) уже есть привычка регулярных партий, (4) географически концентрированы (две агломерации = быстрый и дешёвый CustDev) |
| **Размер beachhead-сегмента** | ≈ 80 тыс. активных хостов (оценка по числу подписчиков 30+ тематических каналов и сообществ по настолкам в МСК/СПб) |

### 1.2 Сценарий и first value moment

| Параметр | Значение |
|----------|----------|
| **Сценарий** | Пятничный вечер, компания 6–8 человек у кого-то дома или в антикафе. Никто не хочет быть ведущим. Хост открывает `aigamemaster.ru`, создаёт комнату, отправляет 6-символьный код в общий чат |
| **Ценностное предложение (UVP)** | **Все играют — никто не ведёт.** Голосовой ИИ-ведущий ведёт партию от первой роли до объявления победителя |
| **First value moment** | Первое голосовое утреннее объявление: «Этой ночью покинул нас… [имя]. Город просыпается в недоумении». Это момент, когда компания понимает «оно работает, и атмосфера есть» |
| **Time to Value (TTV-target)** | ≤ 4 минуты от первого захода хоста на сайт до первого голосового объявления (по факту пилота 9-й недели — 3:50) |

## Шаг 3. Минимальный бизнес-прототип (МБП)

### 3.1 Выбор формата МБП

| Формат | Плюсы | Минусы | Решение |
|--------|--------|--------|---------|
| Лендинг + форма заявки | Быстро тестируется сообщение и канал | Не проверяет ценность голоса (главное отличие) | **Дополнительный** инструмент верхнего слоя воронки |
| Кликабельный прототип Figma | Дешёвый CustDev | Не проверяет реальное прохождение партии | Не подходит — основная ценность в живой партии |
| Concierge (ручной ведущий через Zoom) | Минимальные затраты | Нельзя масштабировать | **Fallback** для антикафе на демо |
| **Веб-приложение (полный MVP)** | Проверяет все 12 гипотез одновременно; нативный канал сегмента; работает на любом устройстве | Требует поддержки и стабильности | **Основной формат МБП** |
| Демо-видео 90 сек | Холодные контакты, антикафе | Маркетинговый, не продуктовый сигнал | Используем для холодного outreach к антикафе |

**Решение по МБП:**

- **Основной**: веб-приложение `aigamemaster.ru` (доработано после ПЗ №18: D1, D2, D3 закрыты).
- **Доп. инструменты**: лендинг + Telegram Ads + демо-видео для антикафе.

### 3.2 Требования к МБП (что должен «уметь»)

| Требование | Цель | Реализация |
|------------|------|------------|
| Полный игровой цикл `создал комнату` → финал | Проверка ценности | Done в MVP |
| 2 голосовых характера на выбор (хост) | Проверка M4 (готовность платить за расширение) | Частично — 1 характер; 2-й в Pro |
| Кнопка «Сыграть снова с теми же» (Rematch FR-11.1) | Проверка M10 (retention 7 дней) | Done — D6 из недели 9 |
| Soft-offer «Pro за 99 ₽» после 2-й партии | Проверка M11 | In progress — простая форма с фиксацией нажатия |
| Аналитика воронки: 7 шагов от Visit до Return | Проверка M5–M7 | Done — Web analytics + кастомные backend events |
| Сбор UTM меток по источнику пользователя | Распределение по каналам | Done — `?utm_source=...` |
| Обход autoplay-policy на Safari/iOS (D2) | Стабильность теста H3 | Done после ПЗ №18 |

### 3.3 Целевое действие и собираемые данные

| Целевое действие | Как фиксируется | Что собираем |
|------------------|------------------|---------------|
| Заход на сайт | Web analytics | UTM, время, device hash |
| Регистрация / login | Backend events | user_id, email-домен, время |
| Создание комнаты | Backend events | room_id, host_user_id, время |
| Завершение партии | Backend events | room_id, число участников, длительность, исход |
| «Сыграть снова с теми же» | Frontend event | room_id, лаг между партиями |
| Soft-offer Pro нажат | Frontend event | room_id, host_user, источник |
| Заявка на B2B (антикафе) | Лендинг-форма | Организация, локация, контакт, число столов |

| Параметр | План |
|----------|------|

## Шаг 6. Воронка привлечения и активации

### 6.1 Воронка из 7 шагов

| # | Шаг | Событие | Метрика | Как измеряем | Целевое значение |
|---|-----|---------|---------|--------------|-------------------|
| **1** | Контакт / показ | Пост увидел / получил письмо | Reach | Telegram views, sent emails | 100% (база) |
| **2** | Переход / интерес | Клик по ссылке на сайт / лендинг | CTR | UTM | ≥ 4% (B2C), ≥ 8% reply rate (B2B) |
| **3** | Заход на сайт | Уникальные visits | Web Analytics | Web stats | ≥ 80% от перешедших |
| **4** | Регистрация / Auth | Создание аккаунта | Backend event | Backend logs | ≥ 50% от visits |
| **5** | Создание комнаты | Хост запустил сессию | Backend event | Backend logs | ≥ 60% от auth |
| **6** | Завершённая партия (Activation) | Партия без блокирующих ошибок | Backend event | Backend logs | ≥ 60% от created |
| **7** | Возврат группы (Retention M1, 7d) | Повторная партия за 7 дней | Сравнение `room_id` групп | Backend logs | ≥ 30% |
| **8 (доп.)** | Готовность платить | Нажатие soft-offer Pro / реальная заявка на B2B | Frontend / лендинг-форма | Logs | ≥ 30% (B2C) / ≥ 1 LOI на 5 контактов (B2B) |
